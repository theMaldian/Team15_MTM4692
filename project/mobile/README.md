# YTU Assistant — Mobile (Flutter)

Flutter client for the **YTU Assistant Recruitment System**. Talks to the Express
backend in `../backend` (see [`../BACKEND_API.md`](../BACKEND_API.md) for the full
endpoint contract).

- **App name:** `ytu_assistant`
- **Org:** `edu.yildiz.mtm4692`
- **Platforms:** Android, iOS
- **Min SDK:** Flutter 3.19+, Dart 3.3+

## Project structure

```
lib/
├── main.dart                    # ProviderScope + App bootstrap
├── app.dart                     # MaterialApp.router + theme
├── core/
│   ├── theme/                   # app_colors, app_text_styles, app_theme (M3)
│   ├── network/                 # api_client (Dio), api_endpoints, api_exception

│   ├── router/                  # app_router (go_router)

├── features/                    # (empty — auth/jobs/etc. added later)
└── shared/
    ├── widgets/                 # reusable widgets
    └── models/                  # shared DTOs
```

## One-time bootstrap (generates native android/ ios/ folders)

This repo ships the Dart source, `pubspec.yaml`, and config. The platform
folders (`android/`, `ios/`) are generated by the Flutter tool. From this
directory run:

```bash
# Regenerate the native scaffolding in place without touching lib/ files.
flutter create . \
  --project-name ytu_assistant \
  --org edu.yildiz.mtm4692 \
  --platforms android,ios

flutter pub get
flutter analyze
flutter run            # boots to the placeholder home screen
```

> `flutter create .` is non-destructive to existing Dart files — it only adds the
> missing `android/`, `ios/`, and tooling files.

## Backend connection

`AppConstants.baseUrl` resolves automatically:

| Platform | Base URL |
|----------|----------|
| Android emulator | `http://10.0.2.2:3000` |
| iOS simulator / desktop | `http://localhost:3000` |

For a physical device, set the base URL to the host machine's LAN IP. The backend
has **no CORS middleware**, so a Flutter **web** build would need CORS added
server-side first.

## Jobs flow assumptions

1. **`close` is `PATCH /jobs/:id/close`** (not POST) and **`apply` is `POST /jobs/:id/apply` with no body** — per `BACKEND_API.md`.
2. **Poster info is flat, not nested.** Jobs return `professor_id` / `professor_first_name` / `professor_last_name` / `professor_email`. There is **no academic title or faculty** on any jobs payload, so the detail "Posted by" card shows name + email only.
3. **No `has_applied` flag.** Whether the current student already applied is only known on **`GET /jobs/:id`** via `my_application_id` (null = not applied) + `my_application_status`. The active feed (`/jobs/active`) returns neither — so the "Applied" state appears on the **detail** screen, not on list cards.
4. **`/jobs/active` is student-only and supports no query params.** A professor hitting it gets **403**, so the professor "Browse" tab shows a friendly "student-only feed" notice. All category/department/sort **filtering is client-side** (the controller fetches the full active list then filters/sorts in memory).
5. **`job_category` is free-text** in the DB (no enum constraint). `JobCategory.fromString` maps the four canonical values (`course_assistant`, `lab_assistant`, `research_assistant`, `other`) and falls back to `other`; `JobModel.categoryLabel` prettifies unknown raw strings for display. Editing a job whose category isn't one of the four canonical values will re-save it as the closest dropdown choice.
6. **`GET /jobs/:id` is ownership-scoped for professors** — the backend only returns a professor's *own* jobs (others → 404). So a professor reaching a job detail always owns it; the "non-owner professor → hide action bar" branch is defensive only.
7. **Deadlines are date-only.** Sent as `yyyy-MM-dd`; the date picker enforces a strictly future date (firstDate = tomorrow). Day math is local-timezone based.
8. **Department prefill on the create form** reads `user.department`, which the login payload doesn't include — so it's usually blank until the profile feature hydrates the cached user. Documented, not blocking.
9. **Applicant detail + status updates are deferred.** Tapping an applicant shows a "coming soon" SnackBar; approve/reject (`PATCH /applications/:id/status`) lands with the applications feature.

## Route table

| Route | Screen | Access |
|-------|--------|--------|
| `/` | Splash | public (bootstrap) |
| `/login`, `/register` | Auth | public; signed-in → role landing |
| `/verify-email?email=` | Verify email | public; forced when signed-in & unverified |
| `/forgot-password`, `/reset-password?email=` | Password reset | public |
| `/change-password` | Change password | authenticated |
| `/home` | (redirect) | → `/jobs` (student) or `/my-jobs` (professor) |
| `/jobs` | Job list (Opportunities) | authenticated; data is student-only (prof → notice) |
| `/jobs/new` | Job form (create) | **professor only** |
| `/jobs/:id` | Job detail | authenticated (student or owner professor) |
| `/jobs/:id/edit` | Job form (edit) | **professor only** (owner enforced by backend) |
| `/jobs/:id/applicants` | Applicants | **professor only** (owner enforced by backend) |
| `/my-jobs` | My postings | **professor only** |
| `/profile` | Profile (placeholder) | authenticated |

Shell routes (`/jobs`, `/my-jobs`, `/profile`) render inside `MainShell` with the bottom nav; detail/form/applicants render full-screen over it.

|--------|:------:|:---------:|

| Job detail (`/jobs/:id`) | ✅ (active jobs) | ✅ (own jobs) |

|---------|----|

| 403 on edit/close/applicants (non-owner) | SnackBar "You don't have access to this posting" |

## Auth flow assumptions

1. **Email verification uses a 6-digit numeric code** (not a magic-link token).
   Confirmed in `backend/utils/authUtils.js` (`randomInt(100000, 1000000)`).
2. **Registration does NOT return a token.** The user must verify their email
   and then call `/auth/login` to obtain a JWT. The register screen therefore
   navigates to `/verify-email?email=…` on success, not `/home`.
3. **Faculty and Department are collected on the register screen but NOT sent
   to `/auth/register`.** The backend's `registerUser` only reads
   `email, password, first_name, last_name`. The faculty/department captured at
   sign-up are intended to flow into `PUT /profile` after the user verifies and
   logs in — that wiring lands in the upcoming profile step.
4. **`/auth/resend-verification`** is the canonical resend endpoint (not
   `/auth/resend` — `BACKEND_API.md` wins). 60-second client-side cooldown.
5. **Forms use Flutter's native `Form` + `TextFormField`**, not `reactive_forms`.
   Validators are centralized in `lib/core/validators/auth_validators.dart` so
   they can be plugged into reactive_forms later if the team prefers that API.
6. **JSON models are hand-written** (no `json_serializable` codegen). This keeps
   the project buildable without a `flutter pub run build_runner` step. Field
   names match the backend (snake_case). If you want to switch to codegen later,
   wrap each model with `@JsonSerializable()`, add a `part 'X.g.dart';`
   directive, replace the manual `fromJson`/`toJson` with `_$XFromJson` / `_$XToJson`,
   and run `flutter pub run build_runner build --delete-conflicting-outputs`.
7. **The YTU logo is a text placeholder** (navy rounded square with "YTU" in
   gold). Drop a real logo into `assets/images/ytu_logo.png`, declare it under
   `flutter.assets` in `pubspec.yaml`, and swap `_YtuMark` in
   `features/auth/presentation/widgets/auth_scaffold.dart` for an `Image.asset`.

## Applications & Profile (CV) features

Two feature folders mirror the established jobs-feature conventions (Riverpod
AsyncNotifier/family controllers, manual JSON, go_router, glass-aesthetic
widgets, full TR/EN localization):

- `lib/features/applications/` — student **My Applications** (`/my-applications`:
  list, withdraw pending applications) and professor **Applicant detail**
  (`/jobs/:id/applicants/:userId`: full applicant info + status update).
- `lib/features/profile/` — the real **Profile** screen (replaces the
  placeholder) with the CV sections (skills, languages, experience, education),
  edit screens, and the master catalog (`/skills`, `/languages`).

Bottom nav is role-aware: students see **Fırsatlar · Başvurularım · Profil**,
professors see **İlanlarım · Keşfet · Profil**.

### Applications / Profile assumptions (verified against `BACKEND_API.md`)

1. **Professor status update** uses `PATCH /applications/:applicationId/status`
   and the backend only accepts `approved` / `rejected` — so the status sheet
   offers only those two; `pending` is the read-only initial state (it cannot be
   set back). The `applicationId` comes from the applicants list row.
2. **Applicant detail** is passed the full `ApplicantModel` via go_router
   `extra` (there is no single-applicant GET endpoint); on a cold deep-link it
   falls back to the cached applicants list for that job.
3. **CV sub-resources** (skills/languages/experiences/educations) are shown for
   **students only**. The backend allows them for any role, but they are
   conceptually student CV data, so the professor profile shows the academic
   title and hides the CV sections (per the prompt's guidance).
4. **Skill / language mutations** return the full profile shape; experience /
   education mutations return a single entity. For uniformity, every mutation
   controller simply invalidates `profileControllerProvider` (one refetch).
5. **Faculty/department on profile edit** reuse the `ytu_faculties.dart`
   dropdowns. If a user's stored faculty/department isn't in that static
   snapshot, it isn't pre-selected (and must be re-picked to be saved).
6. Dates are sent as `YYYY-MM-DD`; an empty end date means "ongoing"
   (Devam ediyor / Halen burada).

### Error UX (applications/profile)

`localizedApiError(l10n, error, overrides)` mirrors the auth mapping with
localized strings: 403 → "Bu işlem için yetkin yok", network → "Bağlantı
hatası…", 5xx → generic; other 4xx fall back to the backend `error` text.

## Demo seed data (backend)

To populate the app for a demo, the backend has an **optional, isolated** seed
script (it does not change any API/route behavior):

```bash
cd ../backend
npm install        # if not already
npm run seed       # inserts demo professors, students, jobs, applications
```

It is idempotent (safe to re-run). Demo credentials are written to
`backend/scripts/SEED_CREDENTIALS.md` (gitignored). All seeded users share the
password `DemoPass1` and are pre-verified. Requires the DB to be reachable
(Azure SQL firewall must allow your IP).
